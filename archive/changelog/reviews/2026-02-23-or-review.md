# Protocol Review: O–R Pass
**Date:** 2026-02-23
**Protocols reviewed:** opcua, oscar, pcep, perforce, pjlink, pop3, pop3s, portmapper, postgres, pptp, prometheus, qotd

## Finding 1 — OPC UA `handleOPCUARead` missing SSRF check (HIGH)

**File:** `src/worker/opcua.ts` — `handleOPCUARead()`

**Problem:** `handleOPCUARead` was the only OPC UA handler that did not call `checkIfCloudflare()` before opening a socket. The other three handlers (`handleOPCUAConnect`, `handleOPCUAWrite`, `handleOPCUABrowse`) all had the check. An attacker could send a crafted Read request targeting a Cloudflare IP to bypass the SSRF guard.

**Fix:** Added the standard `checkIfCloudflare` / `getCloudflareErrorMessage` guard block before the socket is opened in `handleOPCUARead`.

---

## Finding 2 — POP3 command injection via username/password CRLF (HIGH)

**File:** `src/worker/pop3.ts` — `sendPOP3Command()`

**Problem:** `sendPOP3Command()` wrote the raw command string to the socket without stripping `\r\n`. The `USER ${options.username}` and `PASS ${options.password}` commands embed user-controlled values directly, so a username of `user\r\nDELE 1` would inject an extra POP3 command on the wire.

The established safe pattern from SMTP and LMTP already handles this:
```typescript
const safeCommand = command.replace(/[\r\n]/g, '');
```
POP3 lacked this line.

**Fix:** Added `const safeCommand = command.replace(/[\r\n]/g, '');` in `sendPOP3Command()` and write `safeCommand` instead of `command`.

---

## Finding 3 — POP3S command injection via username/password CRLF (HIGH)

**File:** `src/worker/pop3s.ts` — `sendPOP3Command()`

**Problem:** Identical to Finding 2. `pop3s.ts` is a TLS variant of `pop3.ts` that shares the same `sendPOP3Command` implementation. The same CRLF injection via `USER ${username}` / `PASS ${password}` was present.

**Fix:** Applied the same `const safeCommand = command.replace(/[\r\n]/g, '');` fix.
