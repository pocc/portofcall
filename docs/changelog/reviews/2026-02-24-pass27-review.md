## Findings

### H-1: `consul.ts` — `atob(parsed.Value)` Returns Latin-1, Corrupts UTF-8 KV Values

**File:** `src/worker/consul.ts:451`
**Impact:** Consul's KV API returns stored values as base64-encoded raw bytes. The handler decoded using `atob(parsed.Value)`, which returns a Latin-1 string — not UTF-8. Any key containing non-ASCII bytes (e.g., `"café"`, `"Привет"`, JSON with Unicode) is returned as corrupted data silently. Example: Consul value `"café"` (UTF-8: `63 61 66 c3 a9`) → base64 `Y2Fmw6k=` → `atob(...)` → `"cafÃ©"` (corruption).

Same bug pattern fixed in `etcd.ts` during the 25th pass (M-10).

**Fix:**
```typescript
const bytes = Uint8Array.from(atob(parsed.Value), c => c.charCodeAt(0));
value = new TextDecoder('utf-8', { fatal: false }).decode(bytes);
```
**Status:** ✅ Fixed

### H-2: `winrm.ts` — `atob(match[1].trim())` Returns Latin-1, Corrupts PowerShell stdout/stderr

**File:** `src/worker/winrm.ts:656,668` (`decodeReceiveStreams`)
**Impact:** WinRM encodes PowerShell stdout and stderr as base64-encoded UTF-8. The handler decoded using `atob(...)` which returns Latin-1, not UTF-8. Any Windows command that outputs non-ASCII text (e.g., locale-specific output, Unicode filenames, non-English system messages) is returned corrupted. Example: PowerShell command `Write-Output "Привет"` → UTF-8 bytes in base64 → `atob(...)` → garbled Latin-1 characters.

**Fix:**
```typescript
const bytes = Uint8Array.from(atob(match[1].trim()), c => c.charCodeAt(0));
stdout += new TextDecoder('utf-8', { fatal: false }).decode(bytes);
```
(Same fix applied to both stdout and stderr paths.)
**Status:** ✅ Fixed
